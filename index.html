1. Introduction

1.1 Objet

Ce projet a pour but d'évaluer les compétences acquises dans le cadre des modules de l'UF "Infrastructure & Système d'information". En groupe de 2, il fallait realiser un projet visant à mettre en place un SI pour une petite entreprise.

Le projet devait proposer une architecture comprenant un pare-feu avec des fonctionnalités telles que DHCP, DNS, filtrage réseau, routage, ainsi que deux clients (un Windows et un Linux) et deux serveurs Linux (un pour la sauvegarde du site web et un autre pour l'hébergement du site web).



1.2 Périmètre

Ce document couvre l'architecture et les fonctionnalités relatives au réseau interne d'une petite entreprise et leurs communications avec le réseau extérieur.

Ce réseau interne doit être en mesure de contenir cinq hôtes et permettre leur interopérabilité. Ces cinq hôtes comprennent deux PC clients, un serveur pfSense faisant office de routeur/pare-feu, un serveur web et un serveur de sauvegarde des fichiers du serveur web.

Dans un souci de sécurité, il est séparé en deux sous-réseaux: un réseau "tampon" DMZ (zone démilitarisée) n'hébergeant que le site web et recevant les communications extérieures, et un réseau interne LAN où se trouvent toutes les autres machines.

Enfin, le site web doit être accessible par tous les membres des réseaux interne (LAN) et externe (WAN).

2. Infrastructure générale

2.1 Topologie du réseau

Il est intéressant de noter que dans le cas d’une virtualisation du réseau, les switches ne sont pas « physiquement » présents dans l’infrastructure, puisqu’ils sont internes au logiciel de virtualisation.



2.2 Définition succincte des éléments des réseaux

2.2.1 pfSense

Le serveur pfSense sert de colonne vertébrale à l’architecture. Il gère les deux réseaux de l’entreprise (LAN et DMZ) ainsi que leurs relations respectives et avec l’extérieur (Internet). Il a les rôles de routeur, serveur DNS, serveur DHCP, Pare-feu, Proxy inverse et passerelle de sortie pour chacun des deux sous-réseaux qu’il gère.


2.2.2 Réseau dédié au serveur web (DMZ)

Ce réseau n’a pour but que l’hébergement du serveur contenant le site web. Par mesure de sécurité, il n’est pas en mesure de communiquer directement avec le réseau interne, ni capable d’accéder à internet.


2.2.3 Réseau Interne (LAN)

Le LAN constitue le réseau interne de l’entreprise. Il contient les deux PC dits « clients », et le serveur de restauration. Ce serveur a pour but de faire des sauvegardes et d’être en mesure de restaurer les fichiers du site web, situé dans le réseau DMZ.

Cependant, cette configuration n’étant pas supportée de manière universelle (notamment sur le virtualiseur VMWare, pour n’en citer qu’un), un masque de sous-réseau à 29 bits sera préféré ici.



2.4 Services permettant la prise en charge des communications réseaux

2.4.1 Distribution des addresses IP

Le serveur pfSense agissant en qualité de routeur, il est notamment responsable de la liaison adresses IP – MAC (table ARP) de chacun des deux serveurs.

Si certaines adresses IP doivent être fixes pour conserver la stabilité de l’infrastructure et des bonnes communications, pfSense peut jouer le rôle de serveur DHCP et attribuer dynamiquement les adresses IP restantes (constituant la pool DHCP) à tout nouvel appareil tentant d’accéder au réseau.


2.4.1.1 Réseau Interne (LAN)
LAN(em2)
pfSense (statique)* 10.10.0.1
Serveur de restauration (statique) 10.10.0.2
Pool DHCP 10.10.0.3 - 10.10.0.6

2.4.1.2 Réseau dédié au serveur web (DMZ)
DMZ(em1)
pfSense (statique)* 10.10.0.9
Serveur web (statique) 10.10.0.10
Pool DHCP potentielle (désactivée) 10.10.0.11 - 10.10.0.14

NOTE : Les deux adresses notées par un astérisque (*) sont réservées automatiquement dès leur attribution au pare-feu pfSense.


2.4.2 Serveur DNS

Le serveur pfSense joue également le rôle de serveur DNS (Domain Name System), permettant la liaison entre un nom de domaine « compréhensible » et l’adresse IP lui correspondant. Ce serveur DNS permet notamment de réaliser des redirections de port (« Port-Forwarding »), qui seront utilisées pour accéder au serveur web depuis le réseau interne (LAN) de l’entreprise, ou le réseau externe (WAN).



2.5 Services permettant de gérer la sécurité des réseaux

2.5.1 Pare-feu

Des règles de pare-feu devront être mises en place par le serveur pfSense pour garantir une base de sécurité suffisante entre les deux sous-réseaux de l’entreprise (LAN et DMZ) et l’extérieur (WAN, donc Internet)

2.5.2 Proxy Inverse

Enfin, en guise de mesure supplémentaire de sécurité autour du serveur web, un proxy inverse (« Reverse-Proxy ») sera mis en place par le serveur pfSense, permettant dans un premier temps de protéger son adresse IP d’origine aux utilisateurs qui le parcourront.

Dans un second temps et si la fréquentation du site le nécessite, le proxy inverse pourra être configuré pour faire de la mise en cache de données (et améliorer la rapidité du site web), ou de l’équilibrage de charge des serveurs à échelle mondiale (« Global Server Load Balancing »), évitant la surcharge du site.

4. Configuration des hôtes

4.1 Serveur pfSense

4.1.1 Configuration des interfaces

Une fois le système installé, il faut assigner chaque interface au bon NIC et leur attribuer une IP. L’IP du WAN étant souvent attribuée par DHCP, sa valeur sera nécessairement différente de celle ci-dessous. Cependant, les adresses IP LAN et DMZ doivent être celles renseignées dans ce document.

WAN Variable
LAN 10.10.0.1/29 (masque 255.255.255.248)
OPT1 / DMZ 10.10.0.8/29 (masque 255.255.255.248)

Tant que le port par défaut du configurateur web est inchangé, il sera accessible sur le port 443 (HTTPS) de l’IP pfSense correspondant au réseau qui contient la machine.

NOTE : Tant que les règles de pare-feu sont inchangées, il peut être nécessaire de réaliser la commande `pfctl -d` dans le shell pour désactiver temporairement le pare-feu.


4.1.2 Serveur DNS

Le serveur DNS s’active dans `Services > DNS Resolver > General Settings`. Par défaut, il sera fonctionnel et actif sur les deux sous-réseaux.

Une fois actif, tous les hôtes du sous-réseau auront le DNS principal à l’IP du pfSense située dans leur réseau, identique à la passerelle de sortie (10.10.0.1 dans le LAN et 10.10.0.9 dans le DMZ).


4.1.3 IP Statiques et serveur DHCP

4.1.3.1 Serveur interne (LAN)

Le serveur DHCP doit être actif sur l’interface LAN dans `Services > DHCP Server > LAN`, avec la range correspondante (voir partie 2.4.1.1). Le serveur DNS et la passerelle doivent être l’adresse du pfSense dans le LAN, soit 10.10.0.1.

Enfin, il faut ajouter la machine du serveur web aux IP statiques du DHCP pour la lui réserver.


4.1.3.2 Serveur dédié au site web (DMZ)

Par défaut, le réseau DMZ n’a pas vocation à distribuer des adresses IP par DHCP. Si toutefois cette option était envisagée, il est possible de reproduire la même configuration que pour le LAN, avec les valeurs du DMZ (DNS/Passerelle à 10.10.0.9 et Range suivante celle décrite dans le tableau de 2.4.1.2).

4.1.5 Port-Forwarding (accès au site web depuis le LAN et le WAN)

Cette étape doit être réalisée après la mise en place du serveur web. Celui-ci doit déjà être en fonctionnement sur le port 80 de la machine, qui doit pouvoir y accéder par son http://localhost ou http://10.10.0.10.

Pour rendre accessible le site depuis le LAN ou WAN, il faut que le DNS «intercepte» une requête vers un port spécifique du serveur (ici le port HTTP de l’adresse IP du pfSense, en LAN et en WAN) et la redirige vers une machine qui est normalement inaccessible à ces réseaux (ici le serveur web, dans le DMZ isolé des autres réseaux).

Cette règle se met en place dans `Firewall > NAT > Port Forwarding`:

Désormais, le site doit être accessible sur le port 80 de l’adresse IP du pfSense (WAN) dans le WAN, et l’adresse IP du pfSense (LAN, 10.10.0.1:80) dans le LAN.

4.2.2 Mise en place du serveur Web

Une fois toutes les dépendances logicielles installées (avec `sudo apt install` + le nom du package), le package Apache2 devrait être disponible sur la machine.

Pour lancer le serveur au démarrage, il faut réaliser la commande `sudo systemctl enable apache2`.

Le contenu du dossier `/var/www/html` doit s’afficher sur le port 80 du localhost, et si le pfSense a été entièrement configuré, sur le port 80 des IP LAN et WAN du pfSense.


4.2.3 Mise en place des fichiers de restauration

Les dossiers suivants et leurs permissions doivent être mis en place en sudo sur la machine:

NOTE : Le fichier restore.sh demande en argument le nom de l’archive à utiliser pour la restauration.

Web server - restore.sh 
#!/bin/bash
# Checks arguments
if [ $# -lt 1 ]; then
echo 1>&2 "ERREUR: Argument manquant (nom du fichier à utiliser, situé dans le dossier ../temp)"
exit 2
elif [ $# -gt 1 ]; then
echo 1>&2 "ERREUR: Trop d'arguments (nom du fichier à utiliser seulement)"
exit 2
fi

# Takes first argument as filename
filename=$1
# If said file doesn't exist:
if [ ! -f "/home/web-server/web_backups/temp/${filename}" ]; then
echo "ERREUR: Le fichier mentionné n'existe pas"
exit 2
fi

rm -rf /var/www/html/* /var/www/temp/*
tar xfz /home/web-server/web_backups/temp/$filename -C /var/www/temp/ # Uncompresses the tar file
mv /var/www/temp/backup/* /var/www/html/ # Moves the uncompressed files from the backup folder to the html folder
rm -rf /var/www/html/temp /home/web-server/web_backups/temp/* # Removes the temp folders

Backup Server - backup.sh 
#!/bin/bash
echo "RETRIEVING DATA FROM WEB SERVER... (10.10.0.10)"
sshpass -p "root" rsync -az -e ssh web-server@10.10.0.10:/var/www/html/ /home/backup-server/web_backups/backup/
if [ "$?" -eq "0" ]
then
    cd /home/backup-server/web_backups/
    tar cfz /home/backup-server/web_backups/backup-$(date +"%y-%m-%d_%H-%M-%S").tar.gz backup/
    mv /home/backup-server/web_backups/backup-$(date +"%y-%m-%d_%H-%M-%S").tar.gz /home/backup-server/web_backups/backups/backup-$(date +"%y-%m-%d_%H-%M-%S").tar.gz
    rm -rf /home/backup-server/web_backups/backup-$(date +"%y-%m-%d_%H-%M-%S").tar.gz
    echo "Successfully created backup data."
else
    echo "ERROR: Something went wrong when trying to communicate with the web server."
fi

rm -rf ./backup/
Backup Server - restore.sh 
#!/bin/bash
# Retrieves the list of all the files in the backups folder.
files=($(find ../backups/ -mtime -8 -type f -exec basename {} \;))

# If there's no file:
if [ -z "$files" ]; then
   echo "No backup files available for restore."
   exit 1
fi

# Else:

# Orders the file array elements by name:
IFS=$'\n' files=($(sort <<< "${files[*]}"))
unset IFS

echo "Backup files available for restore:"
i=0
while [ $i -lt ${#files[@]} ]
do
   echo $i: ${files[$i]}
   ((i++))
done
printf '\n'

echo "Enter the index of the file you want to restore:"
read d
printf '\n'

file=${files[$d]}
# If files[d] doesn't exist:
if [[ ! "$d" =~ ^[0-9]+$ ]] || test  -z "$file" ; then
   echo "ERROR: Invalid value."
   exit 1
fi

# If the file is not in the folder anymore:
#if test -f "/home/backup-server/web_backups/backups/${file}"; then
#echo "ERROR: The file doesn't seem to exist anymore."
#exit 1.
#fi

# Communicates with the web server to send the chosen files:
cd /home/backup-server/web_backups/backups/
sshpass -f ../scripts/pass.txt rsync -R -arz $file web-server@10.10.0.10:/home/web-server/web_backups/temp/

# If error on the communication:
if [ "$?" -eq "0" ]
then
   sshpass -f ../scripts/pass.txt ssh -tt web-server@10.10.0.10 "sudo /home/web-server/web_backups/scripts/restore.sh ${file}"
   if [ ! "$?" -eq "0" ]; then
   echo "ERROR: Something went wrong during the process"
   exit 1
   fi
   echo "Successfully replaced data."
else
   echo "ERROR: Error when trying to communicate with the server."
fi
Backup Server - remove.sh 
#!/bin/bash
if [ -d "/home/backup-server/web_backups/backups" ]

4.4 Clients Windows et Linux (LAN)

Les clients Windows et Linux n'ont pas de configuration particulière spécifiée dans le cahier des charges. Elles n'ont pas d'impératif de nomenclature pour les noms des machines et utilisateurs, ni de logiciels hors ceux spécifiés plus tôt.

Leur réseau doit être configuré automatiquement, par DHCP. Elles récupéreront ainsi une IP libre de la pool DHCP du LAN.
